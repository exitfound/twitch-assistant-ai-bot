# BOT.md — техническая документация

Подробное описание архитектуры, конфигурации и внутренней логики бота.

---

## Жизненный цикл бота

### Запуск

Точка входа — `bot.py`, функция `run_bot()`. Бот запускается через `asyncio.run()`.

```
asyncio.run(run_bot())
  └─ Bot() — создание экземпляра
     └─ async with Bot() as bot — контекстный менеджер twitchio
        └─ bot.start() — запуск event loop
```

При создании `Bot()` инициализируются:
- `_cooldowns = {}` — словарь `{username: expiry_time}` для per-user кулдаунов (хранит время истечения, не время начала)
- `_bot_name = None` — имя бота (заполняется позже из Twitch API)
- `_channel_id = None` — числовой ID канала стримера (заполняется позже)

### setup_hook (до подключения к Twitch)

1. **`init_db()`** — открывает SQLite-подключение, создаёт таблицы если их нет
2. **Загрузка токена** — если `TWITCH_BOT_TOKEN` и `TWITCH_BOT_REFRESH` заданы в `.env`, добавляет их в twitchio через `add_token()`
3. **Получение ID канала** — `fetch_users(logins=[CHANNEL])` → сохраняет числовой ID стримера в `_channel_id`
4. **Регистрация ChatComponent** — `add_component(ChatComponent(self))`

### event_ready (после подключения к Twitch)

1. **Получение имени бота** — `fetch_users(ids=[bot_id])` → сохраняет ник в `_bot_name`
2. **Подписка на чат и события** — `_subscribe_to_chat()` создаёт `ChatMessageSubscription` + `ChannelFollowSubscription` через EventSub WebSocket
3. **Запуск проактивного цикла** — если `PROACTIVE_ENABLED=true` и `_channel_id` известен, создаёт `asyncio.create_task(_proactive_loop())`

### Подписка на события

`_subscribe_to_chat()` подписывается на два типа EventSub-событий:

1. **`ChatMessageSubscription`** — сообщения чата (обязательно, `as_bot=True`)
2. **`ChannelFollowSubscription`** — новые фолловеры (`moderator:read:followers` scope, `as_bot=True`)

### Завершение

`Ctrl+C` → `KeyboardInterrupt` → блок `finally` в `run_bot()` → `close_db()` закрывает SQLite-подключение.

`SIGTERM` (от `kill`, `systemd stop`, `docker stop`) → `loop.stop()` через `signal handler` → корректное завершение через `finally` → `close_db()`.

`SIGKILL` — не перехватывается (OS-level). WAL может остаться несинхронизированным, но SQLite корректно обрабатывает это при следующем подключении.

## Обработка сообщений (ChatComponent.event_message)

### Шаг 1: Фильтрация собственных сообщений

Бот игнорирует свои сообщения. Сравнение по числовому `chatter.id`.

### Шаг 2: Проверка готовности

Если `_bot_name` неизвестен (`event_ready` ещё не отработал) — сообщение игнорируется.

### Шаг 3: Сохранение в БД

**Каждое** сообщение (кроме ботовых) сохраняется в `chat_messages` + `chat_fts` (через триггер). Безусловно, даже если не адресовано боту.

### Шаг 4: Определение триггера

1. **Mention** — `@botname` в тексте (регистронезависимо)
2. **Сосур** — regex `сосур\w*` (регистронезависимо, Unicode)
3. **Реплай** — ответ на сообщение бота (`reply.parent_user_id == bot_id`)

Если ни один триггер не сработал — выход.

### Шаг 5: Кулдаун

Проверяется in-memory словарь `_cooldowns` (сбрасывается при перезапуске). **Broadcaster (владелец канала) пропускает кулдаун.** Для остальных: если `expiry_time > now` — бот отвечает оставшимся временем и выходит. Два уровня кулдауна: `COOLDOWN_SECONDS` (обычные сообщения) и `COOLDOWN_COMMAND_SECONDS` (`!ask`, `!summary`, `!who`, `!versus`).

### Шаг 6: Извлечение промпта

Из текста удаляются триггеры (`@botname` и `сосур*`). Промпт приводится к нижнему регистру. Если пусто — используется исходный текст.

### Шаг 7: Команды

Проверяются в порядке: `!help` → `!stat` → `!summary` → `!who` → `!versus` → проверка прав для `!fact`/`!defact` → `!defact` → `!fact` → `!ask`.

Все команды выходят до вызова Gemini с контекстом. Кулдаун `COOLDOWN_COMMAND_SECONDS` ставится на `!ask`, `!summary`, `!who`, `!versus`. На `!help`, `!stat`, `!fact`, `!defact` кулдаун не ставится.

### Шаг 7а: Команда `!summary`

Саммари чата текущей сессии:
1. Кулдаун ставится сразу
2. Загружает до 500 сообщений текущей сессии через `get_recent_chat(session_id, 500)`
3. Если сообщений нет — «нет сообщений», return
4. Вызов Gemini с `GenerateContentConfig`: system instruction = `prompt.txt` + дополнительный блок «РЕЖИМ САММАРИ» (основные темы, ключевые моменты, plain text, до 900 символов). Температура `1.2`. Бот сохраняет свой стиль, но факты точные
5. Постобработка и chunking идентичны `!ask` (markdown strip, до 3 чанков по 450 символов)
6. Сохраняется в `bot_interactions` с `[summary]` в `user_message`

### Шаг 7б: Команда `!ask`

Фактический режим — Gemini без персонажа бота:
1. Кулдаун ставится сразу
2. Вызов Gemini с отдельным `GenerateContentConfig`: инструкция «кратко, plain text, без markdown, до 900 символов». Без `prompt.txt`, без контекстных секций
3. Из ответа удаляются: `**`, `##`, `_`, `` ` ``, буллеты (`-`, `•`), нумерация (`1.`, `2)`)
4. Переносы строк заменяются пробелами
5. Если ответ > 1350 символов — обрезается с `...`
6. Разбивается на куски до 450 символов по пробелам (до 3 кусков)
7. Первый кусок отправляется через `message.respond()` (reply с значком «Чат-бот»)
8. Последующие куски — через `_send_chat_message()` (HTTP API) с паузой 1.5 сек
9. Каждый кусок отправляется в отдельном try/except с логированием
10. Сохраняется в `bot_interactions` с `[ask]` префиксом в `user_message`

### Шаг 7в: Команда `!who`

Досье на юзера:
1. Кулдаун ставится сразу
2. Берёт первое слово после `!who` как целевой ник (остальное игнорируется)
3. Параллельно загружает: факты (`get_relevant_facts`), сообщения за всё время (`get_user_messages`, `CONTEXT_WHO_MESSAGES`), прошлые обращения к боту (`get_user_interactions`, 10 записей)
4. Если данных нет — «не знаю ничего», return
5. Формирует промпт с секциями `[Факты]`, `[Сообщения]`, `[Прошлые обращения]` + инструкция: личность, интересы, стиль, со ссылками на конкретные сообщения, 1-3 предложения, максимум 400 символов
6. Вызов Gemini через `_make_gen_config()` (с `prompt.txt`)
7. Обрезка до 420 символов. CAPS по стандартным правилам
8. Сохраняется в `bot_interactions` с `[who]` префиксом

### Шаг 7г: Команда `!versus`

Баттл двух юзеров:
1. Кулдаун ставится сразу
2. Дедупликация ников через `dict.fromkeys()` — одинаковые ники схлопываются. Если меньше 2 уникальных — «нужны два разных ника»
3. Параллельно загружает для каждого юзера: факты (`get_relevant_facts`), сообщения за всё время (`get_user_messages`, `CONTEXT_VERSUS_MESSAGES`), прошлые обращения к боту (`get_user_interactions`, 10 записей)
4. Если данных нет ни про одного — «нет данных», return
5. Формирует промпт с секциями `[Факты]`, `[Сообщения]`, `[Прошлые обращения]` для каждого юзера + инструкция: описать личность/интересы/стиль каждого, сослаться на конкретные сообщения, выбрать победителя, максимум 420 символов
6. Вызов Gemini через `_make_gen_config()` (с `prompt.txt`)
7. Обрезка до 420 символов. CAPS по стандартным правилам
8. Сохраняется в `bot_interactions` с `[versus]` префиксом

### Шаг 8: Сбор контекста

Четыре запроса параллельно через `asyncio.gather`:

```python
facts, recent_chat, context_results, random_knowledge = await asyncio.gather(
    get_relevant_facts(user, prompt),
    get_recent_chat(session_id, CHAT_MESSAGES),
    search_context(prompt, SEARCH_RESULTS),
    get_random_knowledge(KNOWLEDGE_RANDOM),
)
```

### Шаг 9: Установка кулдауна

Кулдаун ставится **до** вызова Gemini API. Причина: Gemini может отвечать 1–5 секунд, без раннего кулдауна бот начнёт генерировать второй ответ параллельно.

### Шаг 10: Вызов Gemini

Нативный async API `google-genai` SDK. Клиент инициализируется лениво при первом вызове. System prompt читается из `prompt.txt` при каждом вызове (hot-reload).

### Шаг 11: Постобработка ответа

1. Удаление `@username:` / `@username,` из начала ответа (`.lstrip(':,')`)
2. Удаление всех оставшихся `@username` адресата из тела (другие ники не затрагиваются)
3. Обрезка до 450 символов
4. CAPS — если входное сообщение капсом ИЛИ `random() < CAPS_PROBABILITY`. `@упоминания` сохраняют оригинальный регистр (`_caps_preserve_mentions()`)
5. Отправка `@username: <текст>`
6. Сохранение в `bot_interactions`

Если ответ пустой — повторный запрос без `[Язык чата]` и `[Контекст канала]`. Если снова пустой — «не удалось получить ответ».

## Проактивные сообщения

Фоновая задача `_proactive_loop()`, запускается как `asyncio.create_task` в `event_ready`.

### Логика

1. Ждёт `PROACTIVE_INTERVAL_MINUTES` после старта
2. Проверяет `get_recent_chat()` — если чат пустой, ждёт следующий интервал
3. Получает `get_random_knowledge()` для контекста
4. Собирает уникальных пользователей из последних 20 сообщений
5. С вероятностью 50% — обращение к случайному активному пользователю, 50% — общий комментарий
6. Формирует промпт с `[Последние сообщения в чате]` + `[Язык чата]`
7. Вызывает Gemini через `_make_gen_config()` (с system prompt из `prompt.txt`)
8. Обрезает до 450 символов
9. С вероятностью `CAPS_PROBABILITY` — CAPS (с сохранением `@ников`)
10. Отправляет через `_send_chat_message()` (HTTP API, без reply-контекста)
11. Сохраняет в `bot_interactions` с `_proactive_` как username

### Отказоустойчивость

Ошибки логируются через `logger.exception()`, цикл продолжается. Отказ одной итерации не останавливает проактивные сообщения.

## Контекст Gemini

При каждом обращении бот формирует промпт из 4 секций, разделённых `\n\n`. Секции включаются условно — если данные есть.

### Секция 1: Сохранённые факты

**Функция:** `get_relevant_facts(username, prompt)`

Все факты спрашивающего юзера (безусловно) + факты других юзеров, если слова из промпта (>3 символов) совпадают через `LIKE '%слово%'`.

Нет лимита — все факты юзера попадают в контекст.

### Секция 2: Последние сообщения в чате

**Функция:** `get_recent_chat(session_id, limit)`
**Env:** `CONTEXT_CHAT_MESSAGES` (по умолч. `50`)

Последние N сообщений всех юзеров текущей сессии (кроме бота), в хронологическом порядке. Скользящее окно — бот видит текущий контекст разговора.

Сообщения бота в окне отсутствуют — бот не видит свои ответы, чтобы не зацикливаться.

### Секция 3: Контекст канала

**Функция:** `search_context(prompt, limit)`
**Env:** `CONTEXT_SEARCH_RESULTS` (по умолч. `10`)

FTS5-поиск одновременно по двум источникам:
- `knowledge_fts` — база знаний (импортированный лор из Telegram)
- `chat_fts` — вся история Twitch-чата (за все сессии)

Результаты делятся пополам: 5 из knowledge + 5 из chat history. Ранжирование по BM25.

Запрос формируется из слов промпта: слова >3 символов получают prefix-match (`слово*`), объединяются через `OR`. Короткие слова отбрасываются.

### Секция 4: Язык чата

**Функция:** `get_random_knowledge(limit)`
**Env:** `CONTEXT_KNOWLEDGE_RANDOM` (по умолч. `10`)

Случайная выборка из `knowledge` (`ORDER BY RANDOM() LIMIT N`). Не зависит от вопроса. Каждый запрос — уникальный набор.

Назначение: бот впитывает лексику и стиль чата. System prompt инструктирует бота цитировать и переиначивать эти фразы.

### Итоговая структура промпта

```
[Сохранённые факты]           ← если есть факты
user1: факт1

[Последние сообщения в чате]  ← если есть сообщения
user1: сообщение1
user2: сообщение2

[Контекст канала]             ← если FTS нашёл совпадения
запись из knowledge
user3: старое сообщение

[Язык чата]                   ← если knowledge не пуста
случайная фраза1
случайная фраза2

username спрашивает: prompt    ← всегда
```

## System prompt

Хранится в `prompt.txt` в корне проекта. Читается при каждом вызове Gemini через `Gemini.get_system_instruction()` → `_load_prompt()`. Можно редактировать без перезапуска бота (hot-reload).

Отправляется как `system_instruction` в `GenerateContentConfig`.

| Блок | Что задаёт |
|---|---|
| Контекст | Кто бот, где работает, аудитория |
| Характер | Резкий, живой, дерзкий |
| Запреты | Без высокопарности, псевдомудрости, банальных оскорблений |
| Стиль | Конкретно, грубо, можно матом |
| Оригинальность | Неожиданный угол, не быть предсказуемым |
| Хаос (10%) | Иногда игнорировать тему |
| Использование контекста | Как работать с секциями `[Контекст канала]`, `[Язык чата]`, `[Последние сообщения]` |
| Длина | 1–3 предложения, 480 символов максимум |

## База данных

### Подключение

Синглтон через `get_db()`. WAL-режим включается при первом подключении. Одно подключение на весь процесс.

### Таблицы

#### chat_messages

Хранит каждое сообщение из чата (кроме ботовых). Индекс на `session_id`.

#### bot_interactions

Хранит пары вопрос-ответ. Отдельная таблица — бот не видит свои прошлые ответы в контексте, что предотвращает зацикливание. Индекс на `session_id`. Используется для `!stat`. Проактивные сообщения сохраняются с `_proactive_` как username. Префиксы в `user_message`: `[ask]`, `[summary]`, `[who]`, `[versus]`, `[follow]`.

#### facts

Факты пользователей. `UNIQUE(username, fact)` + `INSERT OR IGNORE`. Удаление через `LIKE '%query%'`.

#### knowledge

Импортированный лор. `UNIQUE INDEX` на `content`.

### FTS5-таблицы

```sql
chat_fts USING fts5(username, message, content=chat_messages, content_rowid=id, tokenize='unicode61')
knowledge_fts USING fts5(content, content=knowledge, content_rowid=id, tokenize='unicode61')
```

Синхронизация — триггеры AFTER INSERT/DELETE/UPDATE (6 триггеров). Ручные FTS-вставки не нужны.

### Функция _sanitize_fts_query

```
Вход: «Что за медоед такой?»
1. Удаление спецсимволов: «Что за медоед такой»
2. Фильтр по длине (>3): [«медоед», «такой»]
3. Prefix-match: [«медоед*», «такой*»]
4. Объединение: «медоед* OR такой*»
```

## Конфигурация генерации

### Температура (`GEMINI_TEMPERATURE`)

По умолчанию `1.5`. Высокое значение — бот непредсказуем.

| Значение | Поведение |
|---|---|
| 0.0–0.5 | Предсказуемые ответы |
| 0.5–1.0 | Умеренная вариативность |
| 1.0–1.5 | Высокая креативность |
| 1.5–2.0 | Максимальная хаотичность |

### Thinking budget (`GEMINI_THINKING_BUDGET`)

По умолчанию `0` — thinking выключен. Бот отвечает импульсивно.

| Значение | Поведение |
|---|---|
| `0` | Выключен. Быстрее, дешевле |
| `-1` | Модель сама решает |
| `>0` | Явный бюджет. Продуманнее, но медленнее |

## CAPS-режим

Порог: 85% букв в верхнем регистре, минимум 3 буквы. Если входящее сообщение CAPS → ответ CAPS. Дополнительно: с вероятностью `CAPS_PROBABILITY` (30%) любой ответ переводится в CAPS.

При переводе в CAPS `@упоминания` сохраняют оригинальный регистр: `"ТЕКСТ КАПСОМ @username ЕЩЁ ТЕКСТ"`. Реализация — `_caps_preserve_mentions()`, которая разбивает текст по `@\S+`, uppercasит только промежутки.

## Сессии

```python
@property
def session_id(self):
    return time.strftime('%Y-%m-%d')
```

Вычисляется при каждом обращении. Автоматически меняется в полночь. Не кешируется.

Влияние:
- `get_recent_chat` — только текущая сессия. В полночь окно обнуляется
- `get_session_stats` — статистика текущей сессии
- `get_user_messages` — сообщения конкретного юзера за всё время (без привязки к сессии)
- `get_user_interactions` — прошлые обращения юзера к боту (без служебных `[ask]`, `[summary]` и т.д.)
- `search_context` (FTS) — ищет по **всей** истории, не ограничен сессией

## Follow-события

При новом фолловере бот отвечает случайным сообщением из 3 шаблонов (`FOLLOW_MESSAGES`):

```
{user} ЗАЛЕТЕЛ НА КАНАЛ. СОСУРИТИ, ФИКСИРУЕМ ПРОНИКНОВЕНИЕ
{user} ЗАФИКСИРОВАН В СИСТЕМЕ. ДОБРО ПОЖАЛОВАТЬ В РОДНУЮ ГАВАНЬ
{user} ТЕПЕРЬ В КИТЕЖ-ГРАДЕ. ОБРАТНОЙ ДОРОГИ НЕТ
```

Без вызова Gemini. Сохраняется в `bot_interactions` с `[follow]` как `user_message`.

## База знаний (knowledge)

### Формат лор-файла

Обычный текстовый файл (UTF-8), одна запись на строку. `#` — комментарии.

```
# Мемы
exitfound упал с велика на стриме 15 февраля 2025
носок — прозвище маручи

# Лексика
база — одобрение, аналог «based»
кринж — неодобрение
```

### Правила составления

- **Одна запись = одна мысль.** Каждая строка самодостаточна
- **Конкретность важнее объёма.** Конкретные записи лучше матчатся через FTS
- **Оптимальная длина: 20–150 символов**
- **Избегай дублирования.** Регистр имеет значение для SQLite

### Загрузка

```bash
./venv/bin/python3 bot.py --upload-lore lore.txt              # импорт
./venv/bin/python3 bot.py --upload-lore lore.txt --dry-run    # предпросмотр
./venv/bin/python3 bot.py --upload-lore lore.txt --clear-lore # очистить + импорт
./venv/bin/python3 bot.py --clear-lore                        # только очистить
```

Дубликаты пропускаются. Перезапуск бота не нужен.

## Логирование

Уровень `WARNING`. В лог попадают:
- `logger.warning()` — неудачный FTS-поиск, FTS-миграция, неудачная подписка, пустой ответ Gemini
- `logger.exception()` — ошибки Gemini, ошибки отправки `!ask` чанков, ошибки проактивного цикла
- `logger.info()` — отправка `!ask` чанков (количество, размер)

Обычные операции не логируются.
